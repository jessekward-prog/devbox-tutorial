Headless dev box & VS Code remote workflow

A small dedicated machine you build sitting right in front of it — Zorin OS, VS Code, and Claude Code driving the actual infra work — then graduate to fully headless once everything's proven working. Written for someone doing this for the first time — no prior server experience assumed.

Picture opening a tiny takeout kitchen. You set it up standing right in the kitchen, with an assistant chef doing the actual cooking while you watch and approve — and only hand the keys to remote management once you've confirmed deliveries actually work. No dining room, no front counter, and eventually no need to ever stand in the kitchen again.

  • You do the whole build sitting at the box, with Claude driving the infra work in VS Code's chat panel — not you typing curl commands and YAML by hand.
  • You watch it happen: an assistant chef does the actual cooking while you're standing right there, not a recipe card you follow alone.
  • Only once everything's proven working do you set up remote access and unplug the monitor for good.
  • Only once the kitchen's fully running do you close the front door and start managing it from home instead.
  • git push triggers a Coolify build; the app comes up on its own subdomain through this box's own tunnel.
  • When you save and publish your work, the box notices and puts the finished result on a website address by itself — no extra button to press.
  • Because it's plain x86, every Docker image and install script you'd use on a normal Linux server works here unmodified — no ARM compatibility tax.
  • It's a regular little PC, not a special hobbyist one — so ordinary instructions just work, no translating required.
  • You don't need to already know GitHub, Docker, or Coolify. Once Claude has access, it does the actual mechanics — you make the calls only you can make (your domain, your choices) and approve what needs approving.
  • You don't need to already know any of the kitchen equipment. Once your assistant chef has the keys, they run it — you just decide the things only you can decide.

00 / Before you start

Everything below assumes you're starting with nothing built yet. Have these five things ready before step 01:

Think of it as gathering ingredients before you start cooking — nothing below is hard, but it's annoying to stop halfway through because you're missing one of these.

01 / Hardware

Pick an x86 mini PC, not a Pi

A Raspberry Pi is ARM, and Coolify (the tool that watches your code and deploys it) is built and tested for x86 — ARM support exists but is community-patched and flaky. A mini PC in the same footprint runs the exact same install scripts as any normal Linux server, silently, for a few watts.

Think of it as choosing between a tiny hobbyist toy computer (Raspberry Pi) and a proper little PC. The toy is cheaper, but a lot of grown-up kitchen equipment (software) doesn't fit it well. A real mini PC costs a little more and everything just works, the same way it does on any normal computer.

The RAM is the part that actually matters most — below 8GB, the kitchen starts running out of counter space and everything slows down, even if the stove itself (the CPU) is still fine.

02 / Operating system

Install Zorin OS

Download Zorin OS 17 Core (free) and flash it to a USB stick (balenaEtcher or Rufus, both free). Boot the mini PC from that USB and run the graphical installer — about 15–20 minutes. Once it reboots, log in normally and keep the monitor and keyboard connected — you'll be working right here through the next several steps.

Think of it as giving the kitchen its very first staff member. Plug in a screen and keyboard, run through the install like setting up any new computer, and leave them connected — you'll be working right here for a while yet.

Why a full desktop OS instead of a bare-bones server distro: a headless-only server install is marginally lighter on idle memory, but this tutorial actually uses the desktop for the next several steps, while Claude does the infra work in front of you. Zorin is Ubuntu underneath, so everything below still works identically to a "proper" server distro — you're not giving anything up.
Why not the bare-bones version: a stripped-down "server-only" install skips the very screen you're about to use for the next several steps. This kitchen needs a staff room for a while yet before it can run on autopilot.

06 / Public ingress

Cloudflare Tunnel

This is what lets the outside internet reach an app on the box without opening any ports on your router. Claude installs cloudflared, logs into your Cloudflare account (a browser tab opens right here on the box for you to approve), creates a tunnel, and writes the ingress config — roughly. You don't need to know Cloudflare or networking for this — approving that one browser tab is your whole job here.

Think of it as a P.O. box. People send mail (web traffic) to a public address, and it gets forwarded to your actual house (the box) without anyone knowing exactly where that house is. A browser tab will pop up asking you to approve the connection — that's your one part in this step, nothing technical required.

Watch on YouTube — what Cloudflare itself is, in general (not Tunnel-specific, but useful context if you've never used it).

Claude runs this 
# install cloudflared
curl -fsSL https://pkg.cloudflare.com/cloudflare-main.gpg | sudo gpg --dearmor -o /usr/share/keyrings/cloudflare-main.gpg
echo "deb [signed-by=/usr/share/keyrings/cloudflare-main.gpg] https://pkg.cloudflare.com/cloudflared $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/cloudflared.list
sudo apt update && sudo apt install cloudflared

# opens a browser tab for you to approve
cloudflared tunnel login
cloudflared tunnel create devbox
Claude writes this 
/etc/cloudflared/config.yml
tunnel: <tunnel-id-from-create>
credentials-file: /home/<your-username>/.cloudflared/<tunnel-id>.json

ingress:
  - hostname: myapp.yourdomain.com
    service: http://localhost:3000
  - service: http_status:404
Claude runs this 
cloudflared tunnel route dns devbox myapp.yourdomain.com
sudo cloudflared service install
sudo systemctl enable --now cloudflared
If Claude needs to edit that config again later, it'll restart the service, not reload — reload isn't wired up on this systemd unit. This is exactly the kind of gotcha the starter kit already has written down.
One thing to know: changing this file always means a full restart of the delivery service, never a quick refresh — Claude already knows this from the kit, so it won't get stuck on it.

Bonus / Before you unplug

Consider a GUI fallback: self-hosted RustDesk (optional)

Remote-SSH is a terminal, not a screen — great for editing and running commands, useless if you ever want to actually see the desktop (a GUI installer, a visual glitch, anything that isn't a command line). RustDesk gives you that, self-hosted on this same box, so you're not depending on RustDesk's own public relay servers either.

Think of it as a spare set of eyes into the kitchen, not just a walkie-talkie. VS Code is great for giving instructions; sometimes you actually want to look at the screen yourself.

Watch on YouTube — a quick look at what RustDesk is, from NetworkChuck.

Why not route this through the tunnel like everything else: the Cloudflare Tunnel used throughout this tutorial proxies HTTP(S) traffic — RustDesk's relay needs raw TCP/UDP, which doesn't pass through the same way. Rather than opening ports on your router, the cleaner fix is Tailscale: it builds a private network between just your devices, so RustDesk never needs to be reachable from the open internet at all — zero open ports, anywhere. It's a little bit of extra setup, but it's simple, Claude can handle most of it, and you won't regret having it.
Why this one's a bit different: everything else in this tutorial reaches the box through the sealed delivery tunnel. This one instead gets its own private, invitation-only road between just your devices — nothing about it is reachable from the open internet, ever.

Ask Claude to set it up — this installs Tailscale on the box, joins your private network, and self-hosts the RustDesk relay bound to that private address only:

Prompt 
Install Tailscale on this box and connect it to my Tailscale account
(open a browser tab for me to approve it). Then self-host a RustDesk
relay server (hbbs + hbbr) using Docker, bound only to the box's
Tailscale IP — not the public network. Tell me the Tailscale IP,
server address, and public key I'll need for the RustDesk client on
my other devices.
  1. Install Tailscale on your everyday computer too (tailscale.com/download) and sign into the same account — this puts both devices on the same private network.
  2. Install the RustDesk client, open its network settings, and enter the Tailscale IP + key Claude gave you.
  3. Connect once while the monitor's still attached, to confirm the picture actually shows up before you rely on it.
One boot-order gotcha to know about: Tailscale's network interface can come up a moment after Docker starts, which can make a Tailscale-bound container silently fail to bind its port on reboot — this has happened before on a similar box. The "Docker waits for network readiness" prompt further down this page fixes it; worth running once you've got RustDesk working.
One thing to know: occasionally the private network takes a moment longer to wake up than Docker does after a reboot, which can make RustDesk silently stop working until the next restart. There's a prompt further down this page that fixes this permanently — worth running once RustDesk is up.

Bonus / Harden before you disconnect

Helpful prompts to run before you unplug the monitor

Everything below is optional, but worth five minutes while Claude and a screen are both still right there. Each one is a real failure mode, not a hypothetical — paste any of these into the chat panel as-is.

Think of it as installing smoke detectors and an autopilot before you stop watching the kitchen in person. None of this is required to get going — it's the difference between finding out something broke because a customer complained, versus finding out because the kitchen fixed itself.

Restart anything that crashes, automatically. Docker containers don't restart themselves after a crash unless told to — the simplest possible safety net.

If something falls over, stand it back up. By default, a crashed app just stays crashed until someone notices.

Prompt 
Check every Docker container Coolify is running and make sure each one
has a restart policy of "unless-stopped". Fix any that don't, and tell
me which ones you changed.

Watch a specific app and respin it if it goes unreachable. A container can show "healthy" in Docker while still being unreachable from the outside — this exact thing has happened before on a similar box.

Check the food actually arrived, not just that the kitchen said it left. Sometimes everything looks fine from the inside while nothing's actually getting through.

Prompt 
Set up a systemd timer that checks every 5 minutes whether
https://myapp.yourdomain.com responds. If it doesn't, restart that
app's container automatically and log what happened. Show me the
timer and service files before enabling them.

Watch the tunnel itself, not just the apps behind it. Cloudflare Tunnel already reconnects on ordinary network blips — there's no dynamic-IP problem to solve, since it's an outbound-only connection. The real gap is cloudflared getting stuck in a restart-loop and never recovering on its own, which has happened before from an unrelated DNS issue.

Make sure the delivery van didn't just stop running. The van finds its way back after normal traffic — but it can occasionally stall completely and needs someone to notice and restart it.

Prompt 
Set up a systemd timer that runs every 5 minutes and checks two
things: that the cloudflared service is active, and that each of my
public hostnames actually responds over HTTPS. If either check fails,
restart cloudflared and log what happened so I can see it later.

Rule out a stale DNS resolver before it causes cascading failures. A plain, hand-edited /etc/resolv.conf with a dead nameserver listed first can make every lookup take ~5 seconds — slow enough that services relying on fast startup (like cloudflared) restart-loop for reasons that look unrelated to DNS.

Make sure the phone book isn't broken. If every lookup takes 5 seconds because of a bad first entry, everything downstream looks broken in confusing, unrelated ways.

Prompt 
Check whether /etc/resolv.conf is a proper symlink into
systemd-resolved, not a plain file with a stale or unreachable
nameserver listed first. If it's wrong, fix it and tell me what you
changed.

Make Docker wait for the network before starting containers. If you ever add Tailscale, a VPN, or anything else Docker depends on, containers can start before that interface is ready and silently fail to bind their ports — invisible until you actually test from outside.

Don't open the kitchen before the road to it is finished. If the kitchen starts before its own driveway is built, deliveries quietly fail without anyone noticing right away.

Prompt 
If any of my containers depend on a network interface like Tailscale
or a VPN being up first, set up a systemd unit that makes Docker wait
for that interface before it starts. I don't want port bindings to
silently fail after a reboot.
