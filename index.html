Headless dev box & VS Code remote workflow

A small dedicated machine that runs a full git-push-to-deploy loop — mini PC, run headless day to day, Docker + Coolify + a Cloudflare Tunnel underneath, opened only through VS Code Remote-SSH. Written for someone doing this for the first time — no prior server experience assumed.

Picture opening a tiny takeout kitchen that only ever takes orders online — no dining room, no front counter. This guide sets it up: a kitchen (the little computer), a locked staff door only you can open (secure remote access), a shift manager who starts cooking the moment you update a recipe (auto-deploy), and a delivery service that gets the finished food to anyone on the internet without revealing exactly where the kitchen is (the tunnel). You run all of it from your own laptop, like managing the place over a radio, without ever standing in the kitchen yourself.

  • You won't need a monitor or keyboard day to day — but because it's a full desktop underneath, you always can plug one in.
  • Most days it sits somewhere out of the way with no screen attached — but the option to plug one in is never taken away.
  • You open VS Code, connect to devbox, and you have a full editor, terminal, and Claude Code running on it.
  • You open VS Code on your own laptop like always, and it quietly reaches into the little box in the background — a remote control, not a trip to the kitchen.
  • git push triggers a Coolify build; the app comes up on its own subdomain through this box's own tunnel.
  • When you save and publish your work, the box notices and puts the finished result on a website address by itself — no extra button to press.
  • Because it's plain x86, every Docker image and install script you'd use on a normal Linux server works here unmodified — no ARM compatibility tax.
  • It's a regular little PC, not a special hobbyist one — so ordinary instructions just work, no translating required.

00 / Before you start

What you need in hand first

Everything below assumes you're starting with nothing built yet. Have these four things ready before step 01:

Think of it as gathering ingredients before you start cooking — nothing below is hard, but it's annoying to stop halfway through because you're missing one of these.

  • An everyday computer (Mac, Windows, or Linux) that you already use — this is where VS Code lives and where you'll do all your typing. This is your "walkie-talkie" — not the kitchen itself.
  • VS Code installed on that everyday computer. Download it from code.visualstudio.com if you don't already have it.
  • A free GitHub account — this is where your project's code will be stored and versioned. Think of it as the recipe book the kitchen reads from.
  • A domain name added to a free Cloudflare account (any registrar works — Cloudflare just needs to manage its DNS). This tutorial uses yourdomain.com as the example; swap in your own. This becomes the delivery address people use to find what the kitchen makes.
Where to buy the domain: Porkbun is the pick — prices close to what a domain actually costs, no bait-and-switch renewal hikes (the price you see at checkout is roughly the price you pay every year after), free WHOIS privacy and email forwarding included instead of upsold, and no dark-pattern checkout flow. For cheap: skip .com if you don't need it specifically — .xyz, .dev, and .cc are usually a fraction of the price, and Porkbun's "domain of the day" deal is worth a glance before you buy.
Where to get the address: think of domain sellers like mailbox providers — some make the first year cheap then quietly raise the price every year after, and charge extra just to keep your name off a public directory. Recommended Porkbun charges close to the real cost, tells you upfront what renewal costs, and throws in the privacy protection for free. Also: you don't need a .com — a .xyz or .dev address is usually much cheaper and works exactly the same.

No prior Linux or server experience is assumed — every command below is explained before you run it.

You don't need to already know Linux or "servers." Every command is like a recipe step: copy it exactly, one line at a time, and read the sentence above it to know what it does.

01 / Hardware

Pick an x86 mini PC, not a Pi

A Raspberry Pi is ARM, and Coolify (the tool that watches your code and deploys it) is built and tested for x86 — ARM support exists but is community-patched and flaky. A mini PC in the same footprint runs the exact same install scripts as any normal Linux server, silently, for a few watts.

Think of it as choosing between a tiny hobbyist toy computer (Raspberry Pi) and a proper little PC. The toy is cheaper, but a lot of grown-up kitchen equipment (software) doesn't fit it well. A real mini PC costs a little more and everything just works, the same way it does on any normal computer.

02 / Operating system

Install Zorin OS — headless by choice, not by design

Download Zorin OS 17 Core (free) and flash it to a USB stick (balenaEtcher or Rufus, both free). Boot the mini PC from that USB and run the graphical installer once with a monitor and keyboard plugged in — about 15–20 minutes.

Think of it as giving the box its brain for the very first time. You'll need to plug in a screen and keyboard just this once, like setting up a new computer out of the box.

Why a full desktop OS instead of a bare-bones server distro: a headless-only server install is marginally lighter on idle memory, but permanently gives up the option of ever plugging a screen into this box again without reinstalling. On hardware with 16GB+ RAM that saving is negligible — Zorin (Ubuntu underneath, so everything below still works identically) keeps the door open. You'll run it headless day to day; you're just not locked into it.
Why not the bare-bones version: a stripped-down "server-only" install uses a touch less memory, but shuts the door on ever using a screen with this box again. With how much memory these little PCs have, that trade isn't worth it — this way you keep the option, even if you rarely use it.

Zorin doesn't offer an "enable SSH" checkbox during install like a server distro would — turn it on once you're at the desktop, in a terminal:

Once it's installed, open a terminal (Zorin's app menu → Terminal) and switch on the remote-access service — this is the one command that turns "a computer with a screen" into "a computer you can also reach from your laptop":

sudo apt update
sudo apt install -y openssh-server
sudo systemctl enable --now ssh
Turn off auto-suspend. Desktop distros sleep after a period of inactivity by default — harmless with a monitor attached, but the box won't wake itself up for an incoming SSH connection once it's headless. Settings → Power → Automatic Suspend → Off. Screen blanking is fine to leave on; it doesn't affect SSH.
One thing to know: by default the box will fall asleep if left alone for a while, the same as a laptop — except nothing's there to wake it back up once you've unplugged the screen. Turn that off in Settings → Power before you disconnect for good.

After that, unplug the monitor and keyboard. From here on, everything happens over SSH — and if you ever want to sit down at it directly, you still can, any time.

After that, unplug the screen and keyboard. From this point on, you'll mostly "talk" to this box through your laptop — but the back door's still there if you ever want to walk in.

03 / Network

Static IP + key-only SSH

This step does two things: gives the box a permanent address on your home network, and locks its front door so only your laptop can open it — no password to guess or leak.

Think of it as giving the kitchen a fixed street address instead of one that changes every week, and then installing a lock that only opens for one specific key you own — no combination for anyone else to guess.

  1. In your router's admin page, add a DHCP reservation for the box's MAC address, so it always gets the same local IP. (Simpler and more reliable than configuring a static IP on the box itself.)
  2. If your everyday computer doesn't already have an SSH key, make one:
# on your everyday computer — skip if `ls ~/.ssh/id_ed25519.pub` already shows a file
ssh-keygen -t ed25519 -C "your-email@example.com"
# press Enter through the prompts to accept the defaults

This creates a matched pair of files: a private key that never leaves your laptop, and a public key you're allowed to hand out. The box will only let in whoever holds the private half.

This makes two matching pieces: one you keep secret on your laptop forever, and one you're allowed to give to the box. They only work as a pair — like a lock and its one matching key.

  1. Copy your public key onto the box (you'll still log in with your password this one time):
# from your everyday computer
ssh-copy-id <your-username>@<devbox-ip>
  1. Lock the box down to key-only login, disabling passwords entirely:
# on the box, once you're in over the key
sudo sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart ssh
  1. Give it a short name on your everyday computer so you never type the IP again:
# ~/.ssh/config on your everyday computer
Host devbox
    HostName <devbox-ip>
    User <your-username>
    IdentityFile ~/.ssh/id_ed25519
04 / Container runtime

Install Docker

Think of it as installing standardized shipping containers in the kitchen. Every app you run later gets packed into one of these — same shape every time — so it runs the same way no matter what it's shipped to.

curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
# log out and back in for the group change to take effect
05 / Editor

VS Code Remote-SSH

Think of it as the walkie-talkie step. Your VS Code window stays on your own laptop and looks completely normal, but everything you type and run happens on the little box instead — like flying a drone remotely rather than climbing inside it.

  1. In VS Code on your everyday computer, install the Remote - SSH extension (search the Extensions panel).
  2. Ctrl+Shift+P → Remote-SSH: Connect to Host → devbox (it reads the SSH config you wrote in step 03).
  3. VS Code installs a small helper on the box automatically on first connect — no manual step.
  4. Open a folder on the box. You now have a full editor, integrated terminal, and file explorer running on the mini PC; only the window itself is local.
Extensions don't carry over. Remote-SSH keeps a separate extension set per remote — reinstall the ones you actually use (GitLens included worth it) on the remote side too.
One thing to know: any add-ons you use in VS Code have to be installed a second time for this remote connection specifically — they don't automatically come along for the ride.
06 / Public ingress

Cloudflare Tunnel

This is what lets the outside internet reach an app running on the box without opening any ports on your router. It needs the domain from step 00 already added to your Cloudflare account.

Think of it as a P.O. box. People send mail (web traffic) to a public address, and it gets forwarded to your actual house (the box) without anyone knowing exactly where that house is or needing a hole punched in your front wall.

# install cloudflared
curl -fsSL https://pkg.cloudflare.com/cloudflare-main.gpg | sudo gpg --dearmor -o /usr/share/keyrings/cloudflare-main.gpg
echo "deb [signed-by=/usr/share/keyrings/cloudflare-main.gpg] https://pkg.cloudflare.com/cloudflared $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/cloudflared.list
sudo apt update && sudo apt install cloudflared

# log into your Cloudflare account, then create the tunnel
cloudflared tunnel login
cloudflared tunnel create devbox

Write /etc/cloudflared/config.yml (sudo — this is the file the systemd service actually reads, not the copy in your home directory):

Now tell the tunnel which local address to forward each delivery to — write this file (needs sudo since it's a system file):

tunnel: <tunnel-id-from-create>
credentials-file: /home/<your-username>/.cloudflared/<tunnel-id>.json

ingress:
  - hostname: myapp.yourdomain.com
    service: http://localhost:3000
  - service: http_status:404
cloudflared tunnel route dns devbox myapp.yourdomain.com
sudo cloudflared service install
sudo systemctl enable --now cloudflared
Editing that config again later? Restart, don't reload — sudo systemctl restart cloudflared. The reload signal isn't wired up on this unit.
One thing to know: if you change this file later, you have to fully restart the delivery service for it to notice — there's no quick "refresh" option.
07 / PaaS layer

Install Coolify

Think of it as hiring a shift manager. Point it at a recipe book once, and from then on, whenever you update a recipe, it automatically starts cooking the new version — you don't stand over it supervising every dish.

curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash

The dashboard comes up at http://<devbox-ip>:8000. Open it, finish the setup wizard, then mint an API token from Profile → Keys & Tokens — useful later if you ever want to script deploys instead of clicking through the UI.

Open http://<devbox-ip>:8000 in a browser on your everyday computer, finish the short setup wizard, and that's your shift manager's office — this is where you'll tell it about new recipes going forward.

08 / Source control

Wire up GitHub

This connects the shift manager (Coolify) to the recipe book (your code) so it knows where to pull from. Simplest possible path to start:

Think of it as introducing your shift manager to the recipe book so it knows where to look for updates.

  1. Create a repository on GitHub (public is simplest to start — no keys needed).
  2. Push your project's code to it.
  3. In Coolify, create a new Application → Public Repository → paste the repo's URL.

Once you're comfortable with the loop, switch to private repos: mint an ed25519 keypair, add the public half as a GitHub deploy key, upload the private half to Coolify, and connect via its private-repo flow instead. Not needed for a first pass.

Later, once this feels normal, you can make the recipe book private and give the shift manager a special key instead of leaving it open to anyone — but that's a step for later, not for your first attempt.

09 / Proof

Deploy something trivial, end to end

Before building anything real on this box, prove the whole loop once with a throwaway app:

Think of it as a test order before opening night — make sure a plate can actually travel the full route before you trust it with real customers.

  1. Point Coolify at a small repo (even a static "hello" page).
  2. Set the domain to the hostname you routed in step 06.
  3. Deploy, then open https://myapp.yourdomain.com from your phone on mobile data (not wifi), to confirm it's really public.
If the app talks to Postgres later, append ?sslmode=disable to the connection string — Coolify's bundled Postgres doesn't speak SSL and most database libraries try it by default.
One thing to know: if a future app needs to talk to a database, there's one extra setting the tutorial commands already account for — otherwise the database politely refuses the connection.
10 / Steady state

What a normal day looks like

  1. Open VS Code → Remote-SSH: devbox.
  2. Edit, run, test in the integrated terminal — it's really running on the mini PC.
  3. git push.
  4. Coolify picks it up, rebuilds, redeploys.
  5. Live at its subdomain through this box's tunnel.

No manual deploy step, no monitor, no separate dashboard you have to babysit — the whole loop runs from inside the editor.

Every day looks the same: open your laptop, connect like normal, make changes, save and publish, and a few moments later the update is live on the internet. No extra steps to remember.

11 / Optional

Claude Code, no browser bridge needed

Getting a terminal-based AI assistant into a browser dashboard usually means extra plumbing — a proxy, injected auth, iframe cookies — because the terminal has to render inside a web page. None of that applies here, because this box is only ever opened through VS Code:

Think of it as a sous-chef who can also work the kitchen directly. Since you're already remotely "inside" the kitchen through VS Code, there's no extra door to build for them — they just walk in the one you already opened.

npm install -g @anthropic-ai/claude-code

Install it on the box, then just run claude in VS Code's own integrated terminal — it's already an authenticated SSH session, so there's nothing else to wire up.
